# Tuition Scheduler — Development Guide

A daily timetable generator for a tuition centre (grades 5–10, SSC / ICSE). Part-time
teachers report availability each evening via a Telegram bot; a CP-SAT solver builds the
next day's timetable against hard constraints and soft preferences; management reviews and
publishes it from a web dashboard.

**Stack:** FastAPI + SQLAlchemy 2.0 (async) · NeonDB (PostgreSQL) · Google OR-Tools
CP-SAT · Telegram bot (aiogram or python-telegram-bot) · APScheduler · Next.js.

---

## Guiding principles

1. **Riskiest-first.** The solver is the only novel, high-uncertainty component. Build and
   prove it before any plumbing.
2. **Contract-first.** The solver speaks only `SolverInput` → `SolverResult` (Pydantic
   models). Mock data and the DB both produce `SolverInput`, so the data source can change
   without touching the algorithm.
3. **One validator, two callers.** Hard-constraint validation lives in one module, used by
   both the solver and the UI's drag-drop editor. The UI can never save what the solver
   would reject.
4. **Phase gates.** Each phase has a Definition of Done (a test gate). Don't build on an
   unproven phase. The *one* thing allowed to evolve backward is the `SolverInput` contract
   — expect to refine it once at Phase 2, and version it.
5. **UTC in, IST out.** Store all timestamps in UTC; convert using
   `institution_settings.timezone` (`Asia/Kolkata`).

## Phase 0 — Foundations & scaffolding

**Objective:** stand up the skeleton so every later phase is friction-free; prove DB
connectivity end to end; lock the data contract.

**What you build**
- Python environment and dependency manifest (FastAPI, uvicorn, SQLAlchemy, asyncpg,
  alembic, ortools, pydantic, pydantic-settings, APScheduler, the Telegram lib, plus
  pytest, pytest-asyncio, httpx, and optionally hypothesis for testing).
- A minimal FastAPI app (`main.py`) with a `/health` route.
- Alembic configured for **async** (the generated `env.py` must use the async engine),
  with the first autogenerated migration applied to a real Neon database.
- `seed.py` — a script that inserts a realistic, deterministic mock dataset
  (≈ 6 batches, ≈ 8 subjects, ≈ 20 teachers with qualifications, weekly targets, slots,
  one institution-settings row).
- `schemas.py` — the **`SolverInput` and `SolverResult`** Pydantic models. This is the
  contract; treat it as load-bearing.
- pytest set up; a CI workflow (GitHub Actions) that installs deps and runs the suite.

**Tasks**
1. Initialise the repo, virtualenv, and dependency manifest.
2. Drop in `models.py` and `db.py`; create `config.py` reading `DATABASE_URL`,
   `BOT_TOKEN`, etc. from the environment.
3. `alembic init`, point `target_metadata = Base.metadata`, convert `env.py` to async,
   autogenerate the initial migration, run `alembic upgrade head` against Neon.
4. Write `seed.py` and run it.
5. Define `SolverInput` (batches, slots-for-the-date, available teachers + windows,
   qualifications, per-(batch,subject) remaining weekly target, weights) and
   `SolverResult` (assignments, unfilled slots, solver status, objective value).
6. Add a trivial passing test and wire CI.

**Definition of Done (test gate)**
- `alembic upgrade head` succeeds against Neon and recreates the full schema.
- `seed.py` produces the same known dataset every run.
- `pytest` runs and CI is green.
- `SolverInput` / `SolverResult` are defined and documented in `schemas.py`.

**Pitfalls**
- asyncpg under Neon's PgBouncer needs `statement_cache_size=0` (see `db.py`); without it
  prepared statements break.
- Alembic's default `env.py` is sync — you must adapt it for the async engine, and enum
  types sometimes need manual handling in the first migration.
- Use Neon's pooled (`-pooler`) endpoint in app config; a direct endpoint is fine for
  Alembic but pooled is what the app should use.

## Phase 1 — Solver core (pure, no I/O)

**Objective:** a deterministic `solve(SolverInput) -> SolverResult` using CP-SAT, with a
matching hard-constraint `validator`, exercised entirely against fixtures.

**What you build**
- `scheduler/solver.py`: build CP-SAT booleans only for *possible* (batch, period,
  subject, teacher) combinations (skip unqualified / unavailable — this keeps the model
  small), add the hard constraints, build the weighted objective from soft terms, set a
  solver time limit, and return the result including any unfilled slots.
- `scheduler/validator.py`: given a set of assignments, check every hard constraint and
  return pass/fail with the specific violations. (Reused by the UI later.)
- `scheduler/weights.py`: named, tunable weights for each soft term.
- `tests/fixtures/`: at least an easy case, a tight case (barely feasible), and an
  infeasible case (a batch needs a subject with no qualified + available teacher).

**Hard constraints to encode**
- A teacher is never in two batches at the same wall-clock time.
- A teacher only teaches qualified subjects; only within their available windows.
- Max lectures per teacher per day.
- One subject+teacher per batch-period cell; no overlapping lectures within a batch;
  fixed durations (guaranteed by the slot grid).
- Weekly target is handled as a *debt-weighted objective*, **not** a hard daily rule —
  harden it only near the end of the week.

**Soft objective terms** (each a tunable weight)
- Prefer the subject's owner/primary teacher (covers regular-faculty, continuity, and
  minimise-substitution).
- Honour teacher preferred hours.
- Balance workload across teachers.
- Avoid gaps between a batch's lectures.
- Place difficult subjects in their preferred windows.

**Definition of Done (test gate) — be strict here**
- **Feasibility:** returns a valid schedule for every satisfiable fixture.
- **Hard-constraint battery:** a dedicated test per constraint, each using an *adversarial*
  fixture that would tempt a violation, asserting the violation never appears
  (no double-booking, no unqualified teaching, availability respected, max/day respected,
  no batch overlaps). Property-based tests (hypothesis) over random valid inputs are a plus.
- **Infeasibility:** correctly reports `infeasible` and identifies the unfilled slots when
  no valid assignment exists.
- **Optimisation:** given two valid options, assert the solver prefers the higher-scoring
  one — e.g. owner teacher chosen over a substitute; difficult subject lands in its
  preferred window; preferred hours used; workload balanced. Test each soft term in
  isolation so they don't mask each other.
- **Determinism:** same input → identical output (fix `random_seed`, single worker in
  tests).
- **Performance:** solves within a set budget (e.g. < a few seconds) at realistic scale,
  with the solver time limit enforced.
- **Validator parity:** any schedule the solver returns passes the validator.

**Pitfalls**
- Creating variables for impossible assignments bloats the model and slows solving — filter
  at variable-creation time.
- Treating the weekly target as a hard daily constraint makes low-availability nights
  infeasible. Keep it in the objective.
- CP-SAT is multi-threaded and non-deterministic by default; set the seed and worker count
  for reproducible tests.
- Over-weighting one soft term can silently dominate the rest — that's why you test them
  individually.

## Phase 2 — Data layer, service & API

**Objective:** feed the solver from Neon and persist its results, expose everything over
FastAPI, and add the job runner — without changing the solver.

**What you build**
- `services/loader.py`: for a target date, read available teachers (+ windows), the
  batch_slots active on that weekday, batch_subjects, qualifications, and the **week-to-date
  conducted counts**, then compute remaining target (`target − conducted`) and assemble a
  `SolverInput`.
- `services/persister.py`: write a `Schedule` (+ `ScheduleEntry` rows, with unfilled slots
  as null-teacher entries), set solver status and objective value.
- `services/scheduling.py`: orchestrate load → solve → persist in one transaction-safe flow.
- `api/`: endpoints to generate a schedule for a date, fetch a schedule, submit/read
  availability, and approve/publish. Approve/publish flips status and (later) triggers bot
  notifications.
- `jobs/scheduler.py`: APScheduler jobs that open polls, fire reminders, and run the cutoff
  per the configured (per-batch or global) windows.

**Tasks**
1. Implement loader queries with eager loading (`selectinload`) to avoid N+1.
2. Implement the debt query carefully across the week boundary (respect
   `institution_settings.week_start_day`, in IST).
3. Implement persister and the scheduling service.
4. Build the API endpoints and Pydantic response models.
5. Wire APScheduler; for tests, drive it with short intervals.

**Definition of Done (test gate)**
- **Parity:** seed a DB equal to a Phase-1 fixture, run the service, and assert the
  persisted schedule matches what the pure solver returns for that fixture. (This proves
  the loader builds `SolverInput` correctly.)
- Integration tests for each endpoint (shape + behaviour) using httpx against the ASGI app.
- Debt query verified across a week boundary and with mixed conducted/cancelled entries.
- Scheduled jobs fire at the configured times (tested with compressed intervals).

**Pitfalls**
- The "tomorrow" date must be computed in IST, then matched against UTC-stored data.
- N+1 queries in the loader will dominate latency — eager-load.
- Multi-step persist needs proper transaction handling so a failure doesn't leave a partial
  schedule.
- Neon scales to zero; a job firing after idle may hit a cold start — allow for it.
- **Testing DBs:** use a **Neon branch** per CI run (fast, isolated, throwaway), or a local
  Postgres in CI, or transactional-rollback fixtures. Avoid testing against the dev branch.

## Phase 3 — Telegram bot

**Objective:** collect availability dynamically and deliver the published timetable.

**What you build**
- `bot/webhook.py`: receive Telegram updates at a FastAPI route (use a secret token);
  dispatch to handlers.
- Teacher onboarding: a `/start` flow that links a teacher's `telegram_chat_id`.
- `bot/polls.py`: the nightly poll with inline keyboards — **Available all day**,
  **Pick slots** (a slot/time-range picker that writes `AvailabilityWindow` rows), and
  **Unavailable**. Each response writes/updates `TeacherAvailability` and stamps
  `responded_at`.
- Reminder + cutoff behaviour wired to APScheduler: reminders at each configured offset to
  non-responders; at cutoff, apply the per-teacher default (`is_default = true`) —
  full-timers to their standard availability, part-timers to unavailable.
- `bot/notify.py`: on publish, message assigned teachers their schedule and message
  unassigned teachers an explicit "no class tomorrow". Log every send to `NotificationLog`,
  including failures.

**Tasks**
1. Register the webhook; verify the secret token on every update.
2. Build the poll and slot-picker handlers; persist responses.
3. Connect reminders/cutoff to the scheduler from Phase 2.
4. Implement notification dispatch, triggered by the publish endpoint (so it's testable
   before the UI exists).
5. Handle the multi-batch case: a teacher's effective cutoff is the earliest among the
   batches they could be scheduled into.

**Definition of Done (test gate)**
- Simulating a teacher tapping each option writes the correct rows (full / partial+windows /
  unavailable) with `responded_at` set.
- A non-responding teacher gets the correct default applied at cutoff (`is_default = true`).
- Reminders fire at the configured offsets and stop once the teacher responds.
- Publishing a schedule sends the right messages to assigned and unassigned teachers and
  logs them; delivery failures are recorded, not swallowed.
- Tested with a test bot token and/or a mocked Telegram API.

**Pitfalls**
- Telegram retries updates — handlers must be idempotent.
- Some teachers will have blocked the bot or never run `/start`; surface those rather than
  failing silently.
- Reminder scheduling is timezone-sensitive (IST).
- Mind Telegram rate limits when fanning out notifications.

## Phase 4 — Web UI & integration

**Objective:** the admin dashboard and "overdrive" editor, tying the whole system together.

**What you build**
- Next.js app with auth (login as a `User`) and an API client.
- Master-data CRUD screens (teachers, batches, subjects, qualifications, targets, slots,
  poll windows).
- The timetable grid (FullCalendar resource-timeline, or a custom grid with `dnd-kit`).
  **Every drag-drop calls the validate endpoint** (the Phase-1 validator exposed over the
  API); illegal moves are rejected server-side.
- The review → approve → publish flow (publish triggers the Phase-3 notifications).
- Dashboards: availability status for the night, weekly-target progress, syllabus
  completion.

**Tasks**
1. Scaffold Next.js, auth, and the API client.
2. Build the grid + drag-drop with live server-side validation.
3. Build the approve/publish action.
4. Build the status/progress dashboards.

**Definition of Done (test gate)**
- End-to-end happy path: seed → availability (bot or seeded) → generate → admin edits (an
  invalid move is blocked) → approve → teachers notified.
- The UI reflects DB state after every action.
- **Validation parity:** the UI cannot persist any edit the solver wouldn't allow
  (validation is enforced on the server, not just the client).

**Pitfalls**
- Never trust client-side validation alone — the server decides.
- Plan the optimistic-UI-vs-server-rejection flow for drag-drop.
- Keep the grid in sync after edits and after a re-generation.
- Make the drag-drop keyboard-accessible.

---

## After Phase 4 — cross-cutting

- **One end-to-end integration test** that exercises the full nightly cycle on a Neon branch.
- **Observability:** structured logging around the solver (input size, status, objective,
  solve time) and the bot (sends, failures). These are your first debugging tools in
  production.
- **Deployment:** FastAPI (app + bot webhook + APScheduler) on one host; Next.js separately;
  Neon is managed. Ensure the webhook URL is HTTPS and the job runner has a single active
  instance (don't double-fire reminders across replicas).

|------------------|----------------------------------------|--------------------------------------------|
| Solver           | pytest, hypothesis                     | hard-constraint battery, optimisation, perf|
| Validator        | pytest                                 | parity with solver, every hard rule        |
| Data / service   | pytest-asyncio, Neon branch            | loader parity, debt query, persistence     |

| Bot              | test bot token / mocked Telegram API   | poll responses, defaults, notifications    |
| End-to-end       | full stack on a Neon branch            | the whole nightly cycle                    |
